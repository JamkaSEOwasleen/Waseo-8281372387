### Database Functions

```sql
-- Function to get the next batch of highest-priority items
CREATE OR REPLACE FUNCTION get_next_pseo_batch(p_limit INTEGER DEFAULT 100)
RETURNS TABLE (
  queue_id INTEGER,
  location_slug TEXT,
  location_name_ar TEXT,
  location_name_en TEXT,
  country_code TEXT,
  country_ar TEXT,
  niche_slug TEXT,
  niche_name_ar TEXT,
  niche_name_en TEXT,
  intent TEXT,
  format TEXT,
  target_keyword_ar TEXT,
  target_keyword_en TEXT,
  priority_score DECIMAL
) AS $$
BEGIN
  RETURN QUERY
  SELECT 
    cq.id AS queue_id,
    l.slug AS location_slug,
    l.name_ar AS location_name_ar,
    l.name_en AS location_name_en,
    l.country_code,
    l.country_ar,
    n.slug AS niche_slug,
    n.name_ar AS niche_name_ar,
    n.name_en AS niche_name_en,
    cq.intent,
    cq.format,
    cq.target_keyword_ar,
    cq.target_keyword_en,
    cq.priority_score
  FROM pseo_content_queue cq
  JOIN pseo_locations l ON cq.location_id = l.id
  JOIN pseo_niches n ON cq.niche_id = n.id
  WHERE cq.status = 'pending'
    AND l.is_active = true
    AND n.is_active = true
    AND (cq.scheduled_date IS NULL OR cq.scheduled_date <= CURRENT_DATE)
  ORDER BY cq.priority_score DESC, cq.created_at ASC
  LIMIT p_limit;
END;
$$ LANGUAGE plpgsql;
```

```sql
-- Function to calculate priority score for all pending items
CREATE OR REPLACE FUNCTION recalculate_priority_scores()
RETURNS VOID AS $$
BEGIN
  UPDATE pseo_content_queue cq
  SET priority_score = (
    COALESCE(cq.search_volume, 10)::DECIMAL / 
    GREATEST(COALESCE(cq.keyword_difficulty, 50), 1)::DECIMAL
  ) * (
    SELECT priority FROM pseo_locations WHERE id = cq.location_id
  )::DECIMAL / 100.0
  * CASE 
    WHEN cq.intent = 'commercial' THEN 1.0
    WHEN cq.intent = 'comparison' THEN 0.8
    WHEN cq.intent = 'how-to' THEN 0.6
    WHEN cq.intent = 'informational' THEN 0.4
    WHEN cq.intent = 'navigational' THEN 0.3
    ELSE 0.5
  END
  * CASE 
    WHEN cq.format = 'service-page' THEN 1.0
    WHEN cq.format = 'comparison' THEN 0.8
    WHEN cq.format = 'guide' THEN 0.6
    WHEN cq.format = 'list' THEN 0.5
    WHEN cq.format = 'tool' THEN 0.7
    ELSE 0.5
  END
  WHERE cq.status = 'pending';
END;
$$ LANGUAGE plpgsql;
```

### The Meta-Strategy: WasafSEO Generates Its Own PSEO Pages

This is the most elegant part of the plan. WasafSEO already has:

2. A JSON-LD schema generation system

4. GEO/AIO optimization built into prompts

**The PSEO system calls WasafSEO's OWN API to generate landing page content.**

This means:
- Every PSEO page is a **living demonstration** of the product
- Zero additional AI architecture needed
- Content quality improves automatically as WasafSEO's prompts improve
- The PSEO system costs basically nothing to run

### Generation Flow

```
1. GitHub Action triggers (daily)
2. PostgreSQL query: get_next_pseo_batch(100)   ← 100 pages/day
3. For each of 100 queue items:
   a. Build content brief request:
      - keyword = target_keyword_ar
      - intent = mapped to WasafSEO intent
      - location = country_code
      - website = WasafSEO's own website config
   b. Call WasafSEO's /api/generate (or Deepseek directly)
   c. Parse the response: { brief, schema }
   d. Transform brief into landing page content
   e. Validate quality score (if < 75, regenerate)
   f. Insert into pseo_published_pages
   g. Revalidate Vercel ISR cache
4. Log all results
5. Next day: next 100 items
```

## 🧠 AI TRAINING INSTRUCTIONS & QUALITY CONTROL SYSTEM

This section defines the **hardcoded rules, constraints, and data variable requirements** that must be embedded into WasafSEO's system prompts when generating PSEO pages. These rules prevent the 5 critical failure modes of programmatic SEO: thin content, weak E-E-A-T, keyword cannibalization, orphaned pages, and hallucination.

> **⚠️ CRITICAL RULE**: WasafSEO's existing `src/lib/prompts.ts` (the content system prompt and schema system prompt) must NEVER be modified. Instead, these rules are injected as **additional constraints** appended to the user message for each PSEO page generation call, layered ON TOP of the existing prompts. The two-call architecture (Call 1: content brief → Call 2: JSON-LD schema) is preserved exactly as-is.

---

### 5.1 — Thin Content & Duplication Prevention

#### Problem
PSEO systems fail when every page reads the same. If the only difference between "SEO services in Riyadh" and "SEO services in Jeddah" is the city name swap, Google detects pattern duplication and deindexes the entire cluster.

#### Hardcoded Rule: Variable-Rich Structured Input

Every PSEO page generation call must include a `location_data` JSON object in the user message with **all** of these fields:

```json
{
  "location_name": "مدينة الرياض",
  "location_name_en": "Riyadh",
  "region": "منطقة الرياض",
  "country": "المملكة العربية السعودية",
  "country_code": "SA",
  "population": 7800000,
  "population_rank": 1,
  "is_capital": true,
  "economic_zone": "Central Region / نجد",
  "climate": "صحراوي حار — صيفاً يتجاوز 45°م",
  "major_industries": ["النفط والغاز", "المالية", "الخدمات الحكومية", "التجزئة", "التقنية"],
  "municipal_districts": ["الملز", "العليا", "السليمانية", "النظيم", "الشفا"],
  "local_business_challenges": [
    "ارتفاع تكاليف الإعلان في السوق التنافسي",
    "قلة المواهب التسويقية المحلية المتخصصة",
    "الحاجة إلى التوافق مع رؤية 2030"
  ],
  "unique_local_regulations": "نظام التجارة الإلكترونية السعودي ولائحته التنفيذية",
  "cultural_notes": "تعد الرياض مركزاً تجارياً رئيسياً بفضل موقعها الاستراتيجي وكونها مقر الحكومة",
  "competitor_density": "high",
  "avg_search_volume_cluster": "12,000-18,000 monthly searches"
}
```

**The generation prompt MUST contain this instruction verbatim:**

> "You are generating content for a specific location. You MUST reference at least 3 of the location_data fields naturally within the content body, not just in the H1 or meta description. If the content is identical to what you would write for any other city, it is LOW QUALITY. You MUST differentiate by weaving local economic context, climate realities, municipal processes, and cultural specifics into the body paragraphs. At minimum: mention one local regulation or municipal process, cite one local industry challenge, and reference one climate or cultural factor."

#### Hardcoded Rule: Forced Structural Variance

The prompt MUST include:

> "Do NOT use the same paragraph structure on every page. You MUST vary the content structure based on these rules:
> - **If intent = 'comparison'**: Start with a problem-sentence, THEN present options, THEN give recommendation
> - **If intent = 'how-to'**: Start with urgency/why-it-matters, THEN list steps, THEN common mistakes to avoid
> - **If intent = 'informational'**: Start with definition/context, THEN deep dive, THEN implications
> - **If intent = 'commercial'**: Start with credibility statement, THEN services/offerings, THEN social proof
> - **If intent = 'navigational'**: Start with clear location+service match, THEN what user will find, THEN next steps
>
> Never begin two consecutive paragraphs with the same word. Vary sentence length: mix short (3-5 words) with medium (10-15 words) and long (20-30 words) sentences. Never use the same H2 heading structure across pages in the same niche."

#### Hardcoded Rule: Contextual Grounding

> "For every claim about why a service matters in this location, provide a specific reason tied to the location_data. Do NOT use generic statements like 'SEO is important for businesses in [City].' Instead, write: 'In [City], where [specific local economic condition], businesses that invest in SEO see [specific outcome].' Every paragraph must answer: why does this specifically matter HERE?"

### 5.2 — E-E-A-T (Experience, Expertise, Authority, Trust) Enforcement

#### Problem
Google's Quality Rater Guidelines require demonstrable E-E-A-T for YMYL (Your Money or Your Life) content. Arabic SEO pages about business services, legal requirements, and financial planning are YMYL. Without E-E-A-T signals, pages will not rank.

#### Hardcoded Rule: Brand Identity Injection

Every PSEO page prompt MUST include the following entity definition injected into the system prompt context:

> "BRAND IDENTITY — You are writing on behalf of Wasleen Technical Services L.L.C, operating as WasafSEO (wasafseo.wasleen.com). The entity behind this content is:
> - **Company**: Wasleen Technical Services L.L.C (registered business)
> - **Brand**: WasafSEO — Arabic SEO Content Brief Generator
> - **Expertise**: Technical SEO, Arabic content strategy, GEO/AIO optimization, Schema.org structured data
> - **Platform**: SaaS platform at wasafseo.wasleen.com that generates complete Arabic SEO content briefs with JSON-LD schema
> - **Tone**: Professional, authoritative, data-driven, helpful. Never promotional or salesy. Avoid superlatives like 'الأفضل' and 'الرائد' unless backed by verifiable data.
>
> Every page must implicitly establish E-E-A-T by:
> 1. Citing specific methodologies and frameworks (not just opinions)
> 2. Referencing recognized SEO authorities and standard methodologies
> 3. Including specific, actionable technical details (not generic advice)
> 4. Demonstrating understanding of local market conditions
> 5. Providing context that only a domain expert would know"

#### Hardcoded Rule: Mandate Technical Specifics

The prompt MUST forbid generic/fluffy language:

> "You are STRICTLY FORBIDDEN from using these vague terms without specific technical elaboration: 'جودة عالية' (high quality), 'احترافية' (professional), 'حلول متكاملة' (integrated solutions), 'خبرة واسعة' (extensive experience), 'أفضل الممارسات' (best practices).
>
> INSTEAD, you MUST be technically specific. For example:
> - ❌ 'نقدم خدمات تحسين محركات البحث' → ✅ 'نقدم خدمات تحسين محركات البحث التقنية تشمل تدقيق الموارد (Resource Auditing)، تحسين بنية الروابط الداخلية (Internal Link Architecture)، وتنفيذ بيانات JSON-LD المنظمة'
> - ❌ 'نستخدم أحدث التقنيات' → ✅ 'نستخدم تحليل المنافسين عبر SEMrush وAhrefs، مع تكامل مباشر مع Deepseek API لتوليد الموجزات الآلي'
>
> If discussing physical services (e.g., aluminum fabrication, contracting):
> - Specify the gauge of aluminum used (e.g., 1.5mm, 2.0mm)
> - Specify the type of powder coating (e.g., polyester powder coating, PVDF)
> - Specify joinery techniques (e.g., TIG welding, CNC routing, mechanical fastening)
> - Never say 'مواد عالية الجودة' — say exactly what material and specification"

#### Hardcoded Rule: Include "Friction" Details

> "Do NOT present a frictionless, idealized version of the service. Every page MUST include at least one 'friction detail' — a realistic challenge, common obstacle, or process complexity:
> - Common permitting or approval timelines (municipal approval processes)
> - Material lead times or supply chain considerations
> - Seasonal factors that affect service delivery (e.g., summer heat slowing outdoor work)
> - Common mistakes clients make that the service helps avoid
> - Budget range considerations (with explicit [SOURCE: verify] flag)
>
> Friction details build trust because they demonstrate real experience. A page without friction reads like AI-generated fluff."

### 5.3 — Keyword Cannibalization Prevention

#### Problem
When 500+ pages target similar keyword clusters, they compete against each other. Google cannot decide which page to rank, so it ranks none — or ranks the wrong page. This kills the entire domain's organic traffic.

#### Hardcoded Rule: Strict Intent Mapping Per Template

Each PSEO page template must have a **primary intent** assigned in the database. The generation prompt receives this intent and MUST adhere to it:

```typescript
// pseo_content_queue table schema addition
type PSEOIntent = 'commercial' | 'informational' | 'how-to' | 'comparison' | 'navigational';

interface PSEOIntentRules {
  commercial: {
    forbidden_keywords: ['كيف', 'طريقة', 'خطوات', 'ما هو', 'دليل'];  // how-to/informational terms
    required_h2: ['خدماتنا', 'لماذا تختار', 'أسئلة شائعة'];
    call_to_action: 'اطلب عرض سعر' | 'احصل على استشارة';
  };
  informational: {
    forbidden_keywords: ['احجز', 'اطلب', 'سعر', 'تكلفة', 'عرض'];  // commercial terms
    required_h2: ['ما هو', 'أنواع', 'فوائد', 'تحديات'];
    call_to_action: 'تعرف على المزيد' | 'اقرأ الدليل الشامل';
  };
  how_to: {
    forbidden_keywords: ['سعر', 'أفضل شركة', 'مقارنة', 'خدمات'];  // comparison/commercial terms
    required_h2: ['الأدوات المطلوبة', 'الخطوة', 'نصائح مهمة'];
    call_to_action: 'ابدأ الآن' | 'جرب الأداة';
  };
  comparison: {
    forbidden_keywords: ['كيف', 'ما هو', 'دليل'];  // how-to/informational terms
    required_h2: ['مقارنة', 'الفرق بين', 'أيهما أفضل', 'مزايا وعيوب'];
    call_to_action: 'قارن الخطط' | 'اختر الأنسب';
  };
  navigational: {
    forbidden_keywords: ['كيف', 'مقارنة', 'سعر'];
    required_h2: ['من نحن', 'خدماتنا', 'تواصل معنا'];
    call_to_action: 'تصفح المنصة' | 'سجل الآن';
  };
}
```

The generation prompt MUST include:

> "Your assigned intent for this page is: **{intent}**. You MUST write exclusively for this intent. Do NOT include content elements that belong to other intents. If the user would arrive searching for '{intent}-related query', they must find exactly what they expect. Use only the H2 headings specified for this intent. The forbidden keywords for this intent must NOT appear anywhere in the content."

#### Hardcoded Rule: URL & Title Tag Rules

> "URL SLUG MUST follow this exact pattern:
> /{pillar-en}/{city-en}/{subtopic-en} — never /{city-en}/{pillar-en}/{subtopic-en}
> Example: /seo-services/riyadh/technical-seo-audit  ✅
> Example: /riyadh/seo-services/technical-seo-audit   ❌
>
> URL slug must be clean: no underscores, no query parameters, no stop words (and, or, the, in, for).
> Maximum 4 path segments total. Never exceed 4.
>
> H1 TAG MUST follow this exact pattern:
> - **commercial**: {Service Name} في {City} — {Unique Angle}
> - **informational**: {Topic} في {City}: {Comprehensive Guide Year}
> - **how-to**: كيفية {Action} في {City}: {Number} خطوة عملية
> - **comparison**: مقارنة {Options} في {City}: أيهما أفضل لـ {Need}
> - **navigational**: {Service Name} {City} — {Brand Name}
>
> Title tag MUST contain the EXACT target keyword phrase within the first 50 characters.
> Title tag MUST be unique across ALL pages in the system — check against a uniqueness index.
>
> META DESCRIPTION MUST NOT match the H1 or any H2 heading. It must be a unique sentence summarizing the page's value proposition for {intent} searchers."

### 5.4 — Orphaned Page & Crawl Budget Prevention

#### Problem
100,000 pages that nobody links to are 100,000 pages Google cannot find. Every PSEO page must be part of a connected internal linking graph. Orphaned pages waste crawl budget and dilute domain authority.

#### Hardcoded Rule: Automated Internal Linking with Contextual Anchor Text

The generation prompt MUST include:

> "You MUST include exactly 3 internal links in the content body (not navigation, not footer):
>
> **Link 1** (Contextual — within first 200 words): Link to the hub page for this pillar using branded anchor text.
> Example: 'تقدم **WasafSEO** [services in {City}](/{pillar-en}/{city-en}/) المتخصصة...'
>
> **Link 2** (Related — mid-content): Link to a complementary subtopic within the same pillar and city cluster.
> Example: 'لمزيد من التفاصيل حول [{related subtopic}](/{pillar-en}/{city-en}/{related-subtopic}/)، اقرأ دليلنا الشامل.'
>
> **Link 3** (Hub return — within last 200 words): Link back to the main pillar category page for this niche.
> Example: 'تصفح جميع خدمات [{Pillar Name}](/{pillar-en}/) المتوفرة.'
>
> Anchor text MUST be contextual and descriptive. NEVER use 'انقر هنا' (click here) or 'هنا' (here). The anchor text must describe what the linked page contains."

#### Hardcoded Rule: Hub-and-Spoke Architecture Enforcement

```typescript
// Generation queue logic — enforce that spoke pages only exist if hub page exists
interface HubSpokeRule {
  hub_url: string;           // /{pillar}/{location}/
  spoke_url: string;         // /{pillar}/{location}/{subtopic}/
  hub_must_exist_first: true; // Never generate spoke without hub
  spoke_must_link_to_hub: true;
  hub_must_link_to_spokes: true; // Hub page auto-generates "related" section
}

// Cross-hub linking rule:
// Every hub page at /{pillarA}/{city}/ must contain at least 1 link to
// the hub page /{pillarB}/{city}/ for the same city (different pillar).
// This creates a cross-pillar linking mesh that distributes authority vertically AND horizontally.
```

The generation prompt for hub pages MUST include:

> "You are generating a HUB page — the main entry point for all content in the '{Pillar Name}' category for '{City}'. This page must:
> 1. List and briefly describe all subtopics/services in this pillar for this city (as an overview)
> 2. Link to each subtopic page using descriptive anchor text
> 3. Include at least 1 link to a related pillar's hub page for the same city (cross-pillar link)
> 4. Include at least 1 link to the same pillar's hub page in a neighboring city (geographic cluster link)"
```

#### Hardcoded Rule: Crawl Budget Efficiency

> "Each page MUST respect these length limits:
> - **Short page (navigational/comparison)**: 500-800 words — no more, no less
> - **Medium page (commercial/how-to)**: 800-1,200 words
> - **Long page (informational)**: 1,200-1,800 words
>
> Pages outside these ranges will be flagged and require manual review before publishing.
>
> Internal links per page: exactly 3 in body (as specified above), plus breadcrumb links, plus footer links (not counted).
> No page may have more than 100 outbound links total (including navigation) — this is a Google crawl budget best practice."

### 5.5 — Hallucination Prevention

#### Problem
LLMs confidently generate false information. For PSEO at scale, a hallucination about pricing, dimensions, legal requirements, or competitor claims could be replicated across thousands of pages, destroying credibility and potentially causing legal liability.

#### Hardcoded Rule: Strict Bounds on Numbers

The prompt MUST include:

> "You are STRICTLY FORBIDDEN from generating ANY of the following without an explicit [SOURCE: verify] flag:
>
> **Never generate these without [SOURCE: verify]:**
> - Exact pricing or rates (e.g., '500 ريال', '$49/month') — use ranges with verification flag
> - Specific dimensions or measurements (e.g., '2.5mm thickness', '10-meter height')
> - Legal or regulatory requirements (e.g., 'you must register with X authority within 30 days')
> - Competitor names and specific claims about competitors
> - Historical dates or event-specific claims (e.g., 'founded in 2015', 'launched in 2020')
> - Claims beginning with 'according to a study' or 'research shows' without naming the source
>
> **These ARE safe to generate factually without flagging:**
> - Geography: city names, region names, country names
> - Language: fact that Arabic is spoken in these countries
> - Common knowledge business domains (e.g., 'SEO helps websites rank higher in Google')
> - Instructions and methodology (e.g., 'the steps to conduct a technical SEO audit include...')
> - Cultural and economic context based on location_data (e.g., 'Riyadh is the capital and commercial center')
>
> **If you are unsure whether a stat is accurate:**
> 1. Do NOT make it up
> 2. Replace with a general statement that does not require a specific number
> 3. OR use the format: '[Research suggests] that businesses in {industry} see [SOURCE: verify] improvement in organic traffic within 6 months of implementing technical SEO.'
>
> **Hallucination penalty**: A page flagged for hallucination triggers automatic review of the last 50 pages generated by the same prompt configuration. If 3+ hallucinations are found, the generation pipeline halts until review. This is enforced at the database level — `pseo_generation_logs.hallucination_flagged` field is set to `true` when flagged."

#### Hardcoded Rule: Citation & Source Protocol

> "Every page type has required source citations:
> - **Stats and data points**: At least 1 per 300 words — if cannot source, use [SOURCE: verify]
> - **Expert quotes**: Must always show [REPLACE WITH REAL QUOTE] flag — never fabricate quotes
> - **Legal claims**: Always flagged [SOURCE: verify]
> - **Pricing claims**: Always flagged [SOURCE: verify]
>
> The citation flag format is: `[SOURCE: verify]` — rendered visually in the generated brief so human reviewers can spot unverified claims before publication."

### 5.6 — Template System: How PSEO Pages Use WasafSEO Components

#### Architecture

PSEO-generated pages are NOT standalone websites — they are dynamic routes within the existing WasafSEO Next.js application, rendered by server components that combine:

1. **Static shell** → Existing WasafSEO components (`Navbar`, `Footer`, `Providers`, `Card`, `Badge`, `Button`, `Skeleton`, `ProgressBar`)
2. **Dynamic content** → Content fetched from Supabase `pseo_published_pages` table at request time
3. **ISR caching** → Incremental Static Regeneration (revalidate every 7 days) for performance
4. **JSON-LD schema** → Auto-generated by the PSEO schema renderer component

#### Template Flow

```
User visits /seo-services/riyadh/technical-seo-audit
  → [middleware.ts] Check if URL matches PSEO pattern (no session required — public)
  → [app/[pillar]/[location]/[subtopic]/page.tsx] Server component
    → Fetch content from pseo_published_pages WHERE url_slug = params
    → Render <PSELandingPage> with:
        - <Navbar variant="landing" />  ← existing WasafSEO Navbar, CTA points to /login
        - <PSEOContent content={page.content} />  ← rendered from DB-stored HTML/content JSON
        - <PSECTA plan="free" />  ← CTA linking to /login with ?ref=pseo-{pillar}-{location}
        - <PSESchema schema={page.schema} />  ← JSON-LD rendered inline
        - <Footer />  ← existing WasafSEO Footer
    → Cache with ISR (revalidate: 604800 = 7 days)
```

#### Key Template Rules

1. **Never create new layout components** — reuse `<Navbar>`, `<Footer>`, `<Providers>` from `src/components/layout/`
2. **Never duplicate UI primitives** — reuse `<Card>`, `<Badge>`, `<Button>`, `<Skeleton>` from `src/components/ui/`
3. **PSEO content is stored rendered** — the full HTML is generated by WasafSEO's API and stored in Supabase. The page component renders it via `dangerouslySetInnerHTML` with a sanitized subset of allowed HTML tags (h2-h4, p, ul, ol, li, a, strong, em, blockquote, table, div, span with class)
4. **No 'use client' unless necessary** — PSEO pages are server components. Client interactivity is limited to:
   - CTA buttons (linking to /login — handled by Next.js Link)
   - Analytics tracking (handled by `src/lib/analytics.ts` via Script component)
   - Mobile hamburger menu (already handled by Navbar component)

### 5.9 — Summary: Quality Control Enforcement Matrix

| Concern | Enforcement Mechanism | Detection | Penalty |
|---|---|---|---|
| Thin Content | Variable-rich input + structural variance prompt rule | Automated word count + uniqueness score per cluster | Page flagged for review, not published |
| Duplication | Forced structural variance + forbidden identical H2s | Cosine similarity check against cluster siblings | Page quarantined, queue item retried with new prompt |
| Low E-E-A-T | Brand identity injection + technical specificity mandate | Manual audit (random 1% sample weekly) | Template prompt revised, affected pages regenerated |
| Keyword Cannibalization | Strict intent mapping + URL/title uniqueness index | DB-level unique constraint on title_tag + url_slug | Generation halted for that intent+location combo |
| Orphaned Pages | Mandatory internal links (3 per page) + hub-spoke validation | DB query detects pages with 0 outbound links | Page not added to sitemap until linking verified |
| Hallucination | Number bounds + citation protocol + [SOURCE: verify] flag | Monte Carlo sampling: 0.1% of pages reviewed weekly | Pipeline halt if 3+ hallucinations in last 50 pages |
| Crawl Budget Waste | Strict word count limits + max outbound links | Automated audit of crawl depth + page size | Oversized pages truncated, under-sized pages merged |
